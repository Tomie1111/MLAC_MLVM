# Proposed MLVM Allocation Output Explanation

This document explains the three CSV files generated by the proposed MLVM part of `allocationTest.js`:

- `proposed_task_allocation_report.csv`
- `proposed_user_allocation_report.csv`
- `proposed_vm_allocation_report.csv`

These files describe the same experiment at three different levels:

- Task level: what happened to each request.
- User level: how many tasks/resources each user received.
- VM level: how each VM was used, scaled, and loaded.

## 1. `proposed_task_allocation_report.csv`

This is the most detailed output. Each row represents one task/request handled by the proposed MLAC-MLVM allocator.

### Purpose

Use this file when you want to answer:

- Which user submitted this task?
- What security level and compartment did the task require?
- Was the task allowed or denied?
- If allowed, which VM handled it?
- How many resources did the task receive?
- Did the VM need predictive in-place scaling before allocation?

### Columns

| Column | Meaning |
|---|---|
| `Task_ID` | Sequential task/request number. |
| `User_ID` | User selected for the task, for example `user_L0`. |
| `User_Clearance` | User security clearance level, `L0` to `L5`. |
| `Policy_Level` | Required policy/security level for the task. |
| `Policy_Compartment` | Required compartment, such as `finance` or `research`. |
| `Action` | Requested operation, such as `read`, `write`, `exec`, or `override`. |
| `Task_Difficulty` | Derived task size: `easy`, `medium`, or `hard`. |
| `Requested_Resources` | Resource units needed by the task. Higher means heavier task. |
| `Decision` | Final result: `ALLOWED` or `DENIED`. |
| `Reason` | Why the task was allowed or denied. |
| `VM_ID` | VM selected for execution. `none` means no VM was allocated. |
| `VM_Level` | Security level of the selected VM. |
| `VM_Compartment` | Compartment of the selected VM. |
| `Eligible_VM_Count` | Number of VMs that passed all access/security/resource checks. |
| `Allocation_Score` | Score used to choose the VM. Lower is better. |
| `In_Place_Scaled` | `yes` if the VM capacity was expanded before allocation. |
| `Resource_Allocated` | Actual resource units given to the task. Denied tasks get `0`. |
| `VM_Remaining_Resources_After` | Remaining VM resource capacity after allocation. |
| `Alloc_Status_Key` | Redis key where the allocation decision was written. |

### Decision Reasons

| Reason | Meaning |
|---|---|
| `GRANTED` | Task passed MLAC checks and was allocated to a VM. |
| `PLACEMENT_DENIED` | Security-aware placement rejected the task due to unsafe co-location/risk conditions. |
| `NO_MEMORY_CAPACITY` | Eligible VM existed, but memory was not enough even after in-place scaling limits. |
| `NO_VM_CAPACITY` | Eligible VM existed, but resource capacity was not enough. |
| `NO_CPU_CAPACITY` | Eligible VM existed, but CPU capacity was not enough. |
| `BLP_DENIED` | Bell-LaPadula access rule failed. |
| `USER_COMPARTMENT_DENIED` | User was not allowed for the task compartment. |
| `ATTRIBUTE_POLICY_DENIED` | User/action attributes did not satisfy the policy. |
| `RISK_THRESHOLD_DENIED` | VM risk was above the allowed threshold. |
| `TRUST_THRESHOLD_DENIED` | VM trust was below the required threshold. |

### Example

```csv
0,user_L0,L0,L0,finance,read,easy,2,ALLOWED,GRANTED,vm-01,L0,finance,2,-0.06705,no,2,268,alloc_status_0
```

Interpretation:

- Task `0` was submitted by `user_L0`.
- The task required policy level `L0`, compartment `finance`, and action `read`.
- It was an `easy` task requesting `2` resource units.
- The task was allowed.
- It was assigned to `vm-01`.
- `vm-01` had `268` resource units left after the allocation.
- No in-place scaling was needed for this task.

## 2. `proposed_user_allocation_report.csv`

This file summarizes the allocation result per user.

### Purpose

Use this file when you want to answer:

- How many tasks did each user submit?
- How many tasks were allocated?
- How many resources did each user receive?
- How many requests were denied?
- Which VMs did each user use?

### Columns

| Column | Meaning |
|---|---|
| `User_ID` | User identifier. |
| `Clearance` | User clearance level. |
| `Total_Requests` | Number of tasks submitted by the user. |
| `Allocated_Tasks` | Number of tasks that received a VM. |
| `Allocated_Resources` | Total resource units allocated to that user. |
| `Denied_Requests` | Number of tasks denied for that user. |
| `Hard_Tasks` | Number of hard tasks for the user. |
| `Medium_Tasks` | Number of medium tasks for the user. |
| `Easy_Tasks` | Number of easy tasks for the user. |
| `Resource_Per_Request` | Average resource units allocated per submitted request. |
| `Assigned_VMs` | VMs used by that user. |
| `Allowed_Policy_Levels` | Policy levels successfully accessed by that user. |

### How to Read This

`Allocated_Tasks` is not the same as `Allocated_Resources`.

Example:

```text
user_L3: 158 allocated tasks, 1226 allocated resources
```

This means `user_L3` received fewer tasks than some other users, but those tasks were heavier on average. The user received `1226` total resource units across `158` successful tasks.

`Denied_Requests` does not always mean access control denied the user. It can also mean that the request passed security checks but could not be placed because of memory, CPU, placement, or capacity constraints.

## 3. `proposed_vm_allocation_report.csv`

This file summarizes the final state of each VM after the experiment.

### Purpose

Use this file when you want to answer:

- Which VMs were used?
- How many tasks ran on each VM?
- How full is each VM?
- Which VMs scaled in place?
- What are the final load, queue, risk, and trust values?

### Columns

| Column | Meaning |
|---|---|
| `VM_ID` | VM identifier. |
| `VM_Level` | VM security level. |
| `Compartment` | VM compartment. |
| `Total_Resources` | Final VM capacity after any in-place scaling. |
| `Used_Resources` | Resource units consumed on the VM. |
| `Remaining_Resources` | Free resource units left. |
| `Allocated_Tasks` | Number of tasks placed on the VM. |
| `Scale_Events` | Number of predictive in-place scaling events. |
| `CPU_Available` | Remaining CPU capacity ratio. |
| `Memory_Available` | Remaining memory capacity ratio. |
| `Load` | Normalized VM load. |
| `Queue` | Queue pressure. |
| `Risk` | Security risk score. Lower is safer. |
| `Trust` | Trust score. Higher is better. |
| `Utilization_Rate_%` | Percent of VM resources currently used. |

### In-Place Scaling

The paper's proposed method does not primarily spawn a new VM when capacity is full. It uses predictive in-place scaling, which expands capacity on existing eligible VMs before congestion gets worse.

In this output:

```text
Scale_Events > 0
```

means the VM was expanded in place.

Example:

```csv
vm-02,L1,research,600,441,159,104,4,...
```

Interpretation:

- `vm-02` is an `L1` VM in the `research` compartment.
- Its final capacity is `600`.
- It used `441` resource units.
- It has `159` remaining.
- It handled `104` tasks.
- It scaled in place `4` times.

### VM Utilization

High utilization means the VM is close to full.

Example:

```text
vm-18 utilization = 97.00%
```

This VM is almost full. Future tasks for the same level/compartment may need in-place scaling or may be denied if CPU/memory/resource limits are reached.

Low utilization means the VM was not heavily used.

Example:

```text
vm-19 utilization = 12.00%
```

This VM stayed mostly idle because the allocator only uses VMs that satisfy the policy level, compartment, trust, risk, placement, and resource checks.

## Allocation Logic Behind the Outputs

For every task, the proposed allocator follows this order:

1. Load the policy from Redis.
2. Verify the policy hash.
3. Select the user and action.
4. Estimate task resource demand.
5. Build the feasible VM set using MLAC/security checks:
   - BLP rule
   - policy level
   - compartment
   - action attributes
   - risk threshold
   - trust threshold
   - placement admissibility
   - CPU, memory, and resource capacity
6. If capacity is tight, try predictive in-place scaling on eligible VMs.
7. Score remaining eligible VMs:

```text
Score = w_load * load + w_queue * queue + w_risk * risk - w_trust * trust
```

8. Pick the VM with the lowest score.
9. Write the decision to Redis.
10. Write task, user, and VM summaries to CSV.

## Which File To Use

Use `proposed_task_allocation_report.csv` when you need per-request evidence.

Use `proposed_user_allocation_report.csv` when you need a user-level resource summary.
